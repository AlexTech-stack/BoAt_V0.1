The BoAt Guidebook

A manual for people who want to understand what BoAt is, what it can do, and how to actually use it. It complements — and links to — the more detailed how-tos and architecture documents in the repository.

Work in progress. BoAt is under active development. APIs, configuration, and behavior may change without notice — when this guidebook and reality disagree, trust --help output and the in-repo docs, then please fix the guidebook.

What is BoAt?

⛵

BoAt is a deterministic automotive simulation and testing platform for Software-in-the-Loop (SIL), Hardware-in-the-Loop (HIL), and CI/CD validation pipelines. At its core sits a tick-based simulation gateway (boat_gateway, C++20) that bridges virtual and physical CAN / CAN FD / Ethernet networks, exposes a gRPC API, and is driven by a Python SDK and CLI. Custom node and protocol logic is loaded at runtime as plugins.

In practice, BoAt lets you:

  • Simulate ECU networks — run deterministic, tick-scheduled simulations of CAN/Ethernet traffic, with seeded reproducibility (same seed → bit-identical output).
  • Bridge virtual and real buses — the same gateway drives vcan0 in CI and a PEAK/Kvaser adapter on the bench.
  • Record, edit, and replay traffic — capture live traffic to ASC/BLF/PCAP/PCAPNG, inspect and modify it frame-by-frame, and play it back onto any interface, with filtering and IP/MAC rewriting for Ethernet.
  • Reverse-engineer unknown traffic — the trace analyzers detect cycle times, AUTOSAR counters/CRCs, signal boundaries, DoIP/SOME/IP services, and PDU multiplexing from raw captures.
  • Speak automotive protocols — AUTOSAR-style PDU routing with I-PDU groups and COM signal packing, ISO 15765-2 (CAN-TP) diagnostics, SOME/IP, TCP — all as plugins.
  • Test systematically — a system test runner (boat test), fault injection, metrics, and an event store for post-run analysis.

Core concepts

The gateway

boat_gateway is the single long-running process everything else talks to. At startup it inspects the interfaces you hand it (BOAT_CAN_INTERFACES, BOAT_ETH_INTERFACES): names matching vcan* get a virtual CAN driver, everything else a physical SocketCAN driver. It serves gRPC on 0.0.0.0:50051.

Frames — one type for every bus

All traffic is represented by a single unified frame type (boat.v1.Frame) with a bus_type discriminator: CAN, CANFD, ETHERNET, TCP, PDU. The same type is used on the wire (gRPC FrameService), in trace files, and inside the gateway. That is why one CLI verb (boat frame send / subscribe) covers every bus type.

Plugins — core owns transport, plugins own conversations

The gateway core does exactly one thing with frames: stateless transport on and off the wire. Everything stateful — TCP connections, ISO-TP segmentation, PDU routing schedules, SOME/IP services — lives in .so plugins loaded at runtime. Plugins receive frames via on_frame (filtered to the bus types they declare) and publish via a frame publisher that feeds the core's single FrameSink — the only path any frame takes to reach a bus.

Two plugin managers run concurrently: a simulation-scoped one (loaded per scenario, ticked deterministically by the simulation scheduler, torn down with the simulation) and an always-on node manager (its own tick thread; hosts persistent plugins like CAN-TP and the PDU router that must react to bus traffic even with no simulation running). Plugins take JSON config appended to their path:

BOAT_NODE_PLUGINS=./pdu_router.so,./can_tp.so?{"iface":"vcan0"}

PDU database

A JSON file (schema: boat-platform/config/pdu_db.schema.json) describing messages and signals — names, CAN IDs, bit positions, byte order, scaling, cycle times, E2E protection. It is what the Commander UI, the PDU router, boat db, and the PDU Editor all read and write. You can create one by hand, in the PDU Editor, by converting an industry .dbc file, or by letting the Trace Analyzer reverse-engineer one from a recording.

Traces

A trace is recorded bus traffic. BoAt reads/writes the industry formats (Vector .asc/.blf for CAN, .pcap/.pcapng for Ethernet — and .pcapng for mixed CAN+Ethernet) and has its own internal .trace binary format (length-prefixed boat.v1.Frame records) that the replay engine consumes. See Trace file formats.

Determinism

The scheduler is tick-based and all randomness is seeded. Running the same scenario with the same seed twice produces bit-identical output — this is enforced by a dedicated test (boat_determinism_seed) and is the property that makes replay and CI regression comparison trustworthy.

Requirements

BoAt's gateway builds and runs on Linux (SocketCAN is a Linux subsystem). The Python SDK/CLI and the web tools run anywhere Python ≥ 3.11 runs, but need a gateway to talk to for anything live.

Building the gateway

cd boat-platform
cmake --preset debug          # or: release, asan, tsan, coverage
cmake --build --preset debug

The gateway binary lands at build/debug/src/gateway/grpc_gateway/boat_gateway (substitute the preset name for other builds).

Virtual CAN setup

For development without hardware, create virtual CAN interfaces once per boot:

sudo modprobe vcan
sudo ip link add vcan0 type vcan && sudo ip link set vcan0 up
sudo ip link add vcan1 type vcan && sudo ip link set vcan1 up   # as many as you like
The Launcher UI (port 8086) can create vcan/veth interfaces and start/stop the gateway from the browser — it needs passwordless sudo for modprobe and ip link.

Running the gateway

# Bare gateway on one virtual bus — gRPC serves on 0.0.0.0:50051
BOAT_CAN_INTERFACES=vcan0 ./build/debug/src/gateway/grpc_gateway/boat_gateway

# CAN + Ethernet + plugins (per-plugin JSON config after '?')
BOAT_CAN_INTERFACES=vcan0,vcan1 \
BOAT_ETH_INTERFACES=veth0 \
BOAT_NODE_PLUGINS=./build/debug/src/plugins/pdu_router/pdu_router.so,./build/debug/src/plugins/can_tp/can_tp.so?{\"iface\":\"vcan0\"} \
  ./build/debug/src/gateway/grpc_gateway/boat_gateway

See Environment variables for the full list, and Use real hardware for physical CAN/Ethernet interfaces.

First steps — a 5-minute smoke test

  1. Start the gateway on vcan0 (previous section).
  2. Check it sees the interface: 
    boat frame list-ifaces
  3. In one terminal, subscribe: 
    boat frame subscribe --bus-types can
  4. In another, send a frame: 
    boat frame send --bus-type can --can-id 0x123 --iface vcan0 --data AABBCCDD
  5. The subscriber prints the frame. Cross-check with the kernel's own view: 
    candump vcan0

If step 5 works, the whole chain — CLI → gRPC → gateway → SocketCAN — is healthy. Now start the web UIs (./start_ui.sh) and open the Dashboard to see the same traffic in the browser.

Gateway Web UIs need gateway

These FastAPI single-file apps control or observe a running gateway. Start them all with ./start_ui.sh (stop: ./stop_ui.sh) or individually with python3 ui/<name>.py. They share a navigation bar, so once one is open you can hop between them.

Launcher :8086

Network interface setup (create vcan/veth from the browser), gateway lifecycle (start/stop the binary with chosen interfaces and plugins), and live gateway log output. The natural starting point of a session.

Dashboard :8080

Live CAN and Ethernet frame trace, BoAt bus-signal log, and event log — the "what is happening on my buses right now" view.

Nodes (Control Panel) :8081

Discovers the Python node scripts under boat-platform/nodes/ and starts/stops them as subprocesses with per-node rolling logs — a process manager for simulated ECUs.

Commander :8082

Interactive frame sender. Loads a PDU database and lets you compose messages signal-by-signal (Intel/Motorola packing done for you), or send raw CAN/CAN FD/ Ethernet frames by hand.

Recorder :8083

Records live traffic to file: PCAPNG (default — CAN + Ethernet mixed in one file), ASC, BLF, or legacy two-file PCAP, plus an optional .jsonl sidecar for BoAt bus signals. Manages sessions and serves the finished files for download.

Debug :8084

gRPC traffic inspector — streams every RPC call made against the gateway (method, caller, lifecycle, sizes, duration, status). Not in start_ui.sh; run on demand: python3 ui/debug.py.

Standalone Tools work without gateway

These tools operate on files — trace files and PDU databases — so they work completely offline. A gateway is only needed for optional actions that explicitly talk to one (e.g. "Push to Gateway"). Start them all with ./start_tools.sh (stop: ./stop_tools.sh).

PDU Editor :8087

Create, modify, validate (JSON-schema), and export PDU database files — messages, signals, bit layouts, scaling, cycle times.

Trace Analyzer :8088

CAN-focused analysis of .blf/.asc/.trace/ .pcapng files: per-ID statistics, cycle-time detection, multi-channel duplicate resolution, AUTOSAR counter detection, and application-signal reverse-engineering. Exports a PDU database skeleton and converts traces for the Trace Editor.

Trace Editor :8089

View, filter, edit, insert, and delete individual frames inside a .trace file. Guided builders for UDP/ICMP Ethernet frames. Save to file, export to PCAPNG, or push straight to a gateway. Has its own detailed built-in howto (also at http://localhost:8089/howto).

Eth Analyzer :8090

Ethernet-focused analysis of .pcap/.pcapng captures: EtherType/VLAN histograms, node and topology inventory, TCP session reconstruction, cyclic-vs-event UDP flow classification, DoIP/SOME/IP recognition, AUTOSAR PDU-multiplex detection, CSV export of every table.

dbc2boatjson (CLI)

Self-contained converter from industry .dbc files to BoAt's PDU database JSON. See Build a PDU database.

How-To: Run a simulation

# Create and start a simulation from a scenario
boat sim create --scenario-id my_scenario
boat sim start --simulation-id <id>

# Deterministic stepping (e.g. in tests): advance exactly 500 ticks
boat sim pause --simulation-id <id>
boat sim step  --simulation-id <id> --ticks 500

# Observe and stop
boat sim watch --simulation-id <id>
boat sim stop  --simulation-id <id>

Scenarios declare which plugins the simulation loads; those plugins are ticked by the simulation scheduler and fully torn down at stop, so every run starts from a clean, reproducible state.

How-To: Send & watch frames

# CAN (classic)
boat frame send --bus-type can   --can-id 0x123 --iface vcan0 --data AABBCCDD

# CAN FD — the bus type implies the FD flag, payloads up to 64 bytes
boat frame send --bus-type canfd --can-id 0x123 --iface vcan0 \
  --data 00112233445566778899AABBCCDDEEFF

# Ethernet
boat frame send --bus-type ethernet --ethertype 0x0800 --dst-ip 10.0.0.1 --data AABB

# Subscribe (streaming; Ctrl+C to stop) — one or several bus types
boat frame subscribe --bus-types can
boat frame subscribe --bus-types can,ethernet

GUI alternative: the Commander (port 8082) does the same interactively, including signal-level message composition from a PDU database. The Dashboard (port 8080) is the GUI equivalent of subscribe.

How-To: Record a trace

With the Recorder UI (port 8083)

  1. Choose a format — PCAPNG (default) records CAN and Ethernet into one file; ASC/BLF are CAN-only; legacy PCAP writes two separate single-bus files.
  2. Tick the CAN buses and (for PCAPNG/PCAP) Ethernet interfaces to record.
  3. Optionally include BoAt bus signals as a .jsonl sidecar.
  4. Start Recording; stop it from Active Sessions; download the files from the session history table.

With the CLI

boat trace start --format pcapng --buses vcan0,vcan1 --eth veth0
# ... traffic happens ...
boat trace stop
boat trace status
ASC/BLF recording requires python-can (pip install python-can). PCAPNG and PCAP recording have no extra dependency.

How-To: Replay a CAN trace

Two hard-separated paths exist — pick by need:

boat trace replay (direct)boat replay import + start/stream (server-side)
Bus typesCAN only (.asc/.blf, CAN records of .pcapng)CAN + Ethernet, all formats
TimingClient-paced, ~5-8 ms gRPC round-trip per frameGateway tick timer, no per-frame overhead
ControlCtrl+C onlypause / resume / seek / stop, repeatable without re-upload
Best forQuick ad-hoc replaysEthernet, high speed, long traces, remote gateways
# Direct: channel 1 → vcan0; filter by channel/ID; speed control; loop
boat trace replay recording.blf --buses vcan0
boat trace replay recording.blf --channel 4 --id 0x040,0x0C0 --speed 0.2 --buses can0
boat trace replay recording.blf --loop --buses vcan0

# Server-side: import once, replay many times
boat replay import recording.blf --trace-id demo
boat replay stream --trace demo --buses vcan0
boat replay pause / resume / stop

Channel mapping is positional: trace channel N maps to the N-th name in --buses (1-based; last one is the fallback). Interface targeting is a replay-time decision — the imported trace stores channels, not interfaces, so one import replays onto any hardware.

How-To: Replay an Ethernet pcap

Ethernet replay is server-side only. IP addresses are rewritten at import time, MAC addresses are assigned at playback time:

# Import: rewrite every packet to the given src/dst IPs
boat replay import capture.pcap --trace-id demo \
  --replay-src-ip 192.168.1.1 --replay-dst-ip 192.168.1.100

# Play onto an interface (must be registered via BOAT_ETH_INTERFACES)
boat replay stream --trace demo --eth-iface eth0

# Verify on the wire
tcpdump -i eth0

Power tools, all on import: --ip-map old=new,... (per-address table), --ethertype ipv4,arp, --protocol udp,icmpv6, --src-port/--dst-port, and post-rewrite --ip-filter/--src-ip-filter/--dst-ip-filter. Per-IP MAC assignment is a playback flag: --mac-map 192.168.0.100=02:de:ad:be:ef:01 on stream/start. IPv4/IPv6 with UDP and ICMP are fully rebuilt with recalculated checksums; fragments and IPv6 extension headers are handled.

The complete reference with the exact processing order lives at boat-platform/docs/howto/replay.md.

How-To: Mixed CAN+Ethernet traces with PCAPNG

PCAPNG is the one open format in BoAt's toolbox that carries CAN and Ethernet in a single file (one interface block per bus, one shared timeline) — and it opens directly in Wireshark, which makes it the right format for handing traces to people outside BoAt.

  • Record — Recorder UI, format PCAPNG (the default), or boat trace start --format pcapng --buses vcan0 --eth veth0.
  • Import — boat replay import capture.pcapng --trace-id demo; both bus types land in one server-side trace, replayable together.
  • Export — turn any internal .trace back into an open file: 
    boat replay export /tmp/demo.trace out.pcapng
    or click Export to PCAPNG in the Trace Editor. TCP/PDU frames have no wire representation and are skipped (counted in the report).
  • Analyze — both analyzers accept .pcapng; the Trace Analyzer reads its CAN interfaces, the Eth Analyzer its Ethernet interfaces, each reporting what it skipped.
  • Replay CAN directly — boat trace replay mixed.pcapng --buses vcan0 replays the CAN records and skips the Ethernet ones.

How-To: Edit a trace

  1. Get a .trace file. Any import creates one on the gateway under /tmp/<trace_id>.trace; or convert a .blf/.asc/.pcapng locally via the Trace Analyzer's Convert action.
  2. Open the Trace Editor (port 8089) — the file dropdown auto-populates from /tmp, traces/, and your home directory.
  3. Filter (bus type, iface, CAN ID, time range), then Edit / Insert After / Delete frames. DLC auto-syncs to the payload; CAN FD lengths are validated; guided builders construct UDP/ICMP packets with correct checksums for you.
  4. Save/Save As to disk, Export to PCAPNG for external tools, or Push to Gateway to upload the edited frames for replay: 
    boat replay start --trace <trace_id> --buses vcan0
Push exists because boat replay import only accepts source formats — it cannot re-ingest an already-converted .trace. Push uploads the binary directly via ImportTraceData. Full details, field semantics, and gotchas: Trace Editor howto.

How-To: Analyze a CAN trace (and reverse-engineer signals)

  1. Open the Trace Analyzer (port 8088), enter or browse to a .blf/.asc/.trace/.pcapng file, and run the staged analysis:
  2. Identify messages — per-CAN-ID statistics, cycle-time detection, Cyclic/Spontaneous classification, and multi-channel duplicate resolution (the same ID relayed by a physical gateway onto a second bus is collapsed to its original source).
  3. Find AUTOSAR counters — detects rolling counters (and their companion checksums/CRCs) inside payloads.
  4. Discover application signals — clusters the remaining payload bits into candidate signals with byte order, signedness, and confidence estimates.
  5. Export the result as a PDU database skeleton (JSON) — a starting point you refine in the PDU Editor — or convert the trace to .trace to hand it to the Trace Editor.

How-To: Analyze an Ethernet trace

  1. Open the Eth Analyzer (port 8090) and point it at a .pcap or .pcapng.
  2. Analyze gives you: EtherType/VLAN histograms, MAC/IP node inventory with role hints, UDP flows classified cyclic vs. event-driven, TCP sessions reconstructed with client/server roles, and a DoIP / SOME/IP service catalog.
  3. If AUTOSAR PDU-multiplexed flows are detected, run the second stage to get a per-Header-ID PDU catalog with sending behavior.
  4. Every table exports to CSV.

How-To: Build a PDU database

From a DBC file

python3 tools/dbc2boatjson.py \
    boat-platform/config/pdu_db.schema.json \
    my_vehicle.dbc  my_vehicle.json \
    --bus "Powertrain_CAN" --bus-type CANFD --validate

Messages, signals, multiplexing, value tables, comments, and GenMsgCycleTime all map over; no external DBC library needed. Reference: boat-platform/docs/howto/dbc2boatjson.md.

From scratch or from analysis

Use the PDU Editor (port 8087) to create/edit databases with live JSON-schema validation — or start from the Trace Analyzer's reverse-engineered export and clean it up there.

Inspect from the CLI

boat db list
boat db show --db pdu_db.json
boat db signal-routes --db pdu_db.json --signal MotorSpeed

How-To: Set up PDU routing

Requires the pdu_router.so plugin loaded on the gateway (via BOAT_NODE_PLUGINS).

# Route PDU 0x100 onto vcan0, cyclically every 100 ms
boat pdu route --id 0x100 --transport can --iface vcan0 --send-type cyclic --cycle-ms 100

# Group PDUs and switch them together (AUTOSAR I-PDU groups)
boat pdu group --id 1 --name "Safety" --pdu 0x100 --pdu 0x200 --disabled
boat pdu enable-group --id 1
boat pdu list-groups
boat pdu list-routes

The router supports cyclic / onChange / mixed transmission schedules, COM signal packing (Intel & Motorola), E2E CRC protection, and deadline monitoring.

How-To: Diagnostics with CAN-TP (ISO 15765-2)

Requires the can_tp.so plugin (note the JSON config selecting its interface):

BOAT_NODE_PLUGINS=./build/debug/src/plugins/can_tp/can_tp.so?{\"iface\":\"vcan0\"} ...

# Configure a session (typical OBD tester ↔ ECU addressing)
boat can-tp configure --nsdu-id diag --source-addr 0x7E0 --target-addr 0x7E8

# Send a payload larger than one frame — segmentation/flow control is automatic
boat can-tp send --nsdu-id diag --source-addr 0x7E0 --target-addr 0x7E8 \
  --data 0123456789ABCDEF0123456789ABCDEF

Because CAN-TP runs in the always-on node manager, it answers incoming diagnostic requests even when no simulation is active.

How-To: Drive a power supply, relay, or generator

Beyond buses and frames, BoAt models electrical devices — programmable DC power supplies, relays/contactors, generators, electronic loads, generic I/O. They are not a new bus type: each device lives as named signals on the always-on signal bus following <kind>.<id>.<channel>[.<role>] (e.g. psu.main.voltage.set in, psu.main.voltage.meas out). A typed DeviceService (boat device) gives a device-shaped view over that convention, backed by deterministic virtual models or real SCPI hardware — the same control surface either way.

Virtual power supply + ignition relay

GW=build/debug/src/gateway/grpc_gateway/boat_gateway
P=build/debug/src/plugins

# Devices are signal-domain — no CAN interface needed.
BOAT_CAN_INTERFACES="" \
BOAT_NODE_PLUGINS="$P/device_manager/device_manager.so,\
$P/virtual_psu/virtual_psu.so?{\"id\":\"main\",\"r_load\":3.0},\
$P/virtual_relay/virtual_relay.so?{\"id\":\"kl15\"}" \
  $GW
boat device list                       # discovered devices + channels (*=settable)
boat device set psu.main voltage 24    # setpoint
boat device set relay.kl15 state 1     # close the contact (0 = open)
boat device read psu.main              # voltage=24 V, current=8 A  (24 V / 3 Ω)

The device_manager plugin discovers devices by observing their published measurements, and the DeviceService delegates to it (the same pattern PduService uses for pdu_router). Programmatically, use from boat.device_node import DeviceNode; for quick interactive use without a rebuild, the Python nodes demo/virtual_psu_node.py and demo/virtual_relay_node.py speak the same signals.

Ignition-gated restbus (KL15)

Wire a relay into the restbus so cyclic transmission follows the ignition, like a real vehicle:

python3 demo/restbus_simulator.py config/<db>.json --kl15-relay kl15
boat device set relay.kl15 state 1     # ignition ON  → cyclic frames start
boat device set relay.kl15 state 0     # ignition OFF → bus goes quiet

Record & replay device state

# record (run the gateway from boat-platform/ so boat_events.db lands here)
BOAT_RECORD_BUS_SIGNALS=devrec BOAT_RECORD_BUS_PREFIXES=psu. \
BOAT_NODE_PLUGINS="$P/virtual_psu/virtual_psu.so?{\"id\":\"main\",\"r_load\":3.0}" $GW
# … drive psu.main voltage, then stop the gateway …

# replay the recorded curve back onto the bus (as named signals, not fake frames)
boat replay from-events --sim-id devrec --signal-id psu.main.voltage.meas -s accelerated -m 20

Real hardware over SCPI

scpi_device drives a bench supply / e-load over TCP (the LXI/raw-socket port, commonly 5025) using the same signal contract as the virtual PSU:

BOAT_NODE_PLUGINS="$P/device_manager/device_manager.so,\
$P/scpi_device/scpi_device.so?{\"id\":\"main\",\"host\":\"192.168.0.5\",\"port\":5025}" $GW
boat device set psu.main voltage 5     # → VOLT 5 over SCPI; MEAS:VOLT?/MEAS:CURR? polled back
Physical devices are live-only: HIL-gated, excluded from the determinism seed test, and never a replay target (a replay reconstitutes recorded state into a virtual model). gpio/modbus are future IDeviceDriver backends implementing the same seam.

Details: docs/howto/devices.md and docs/architecture/device-integration-plan.md; system tests in test/Devices.md (run python3 test/test_device_integration.py).

How-To: Use real hardware

Physical CAN (PEAK, Kvaser, gs_usb — anything SocketCAN)

# Bring the interface up yourself (bitrate to match your bus)
sudo ip link set can0 up type can bitrate 500000

# CAN FD
sudo ip link set can0 up type can bitrate 500000 dbitrate 2000000 fd on

# Hand it to the gateway — non-vcan names get the physical driver automatically
BOAT_CAN_INTERFACES=can0,vcan0 ./boat_gateway

# Verify
boat frame list-ifaces
ip -d link show type can

Physical Ethernet

Physical NICs use raw AF_PACKET sockets and need the raw: prefix plus the CAP_NET_RAW capability on the binary:

sudo setcap cap_net_raw+ep build/debug/src/gateway/grpc_gateway/boat_gateway
BOAT_ETH_INTERFACES=raw:eth0 ./boat_gateway
Prefer setcap over running the gateway as root — a root gateway leaves root-owned trace files under /tmp that block later non-root runs. setcap is stored on the binary's inode, so reapply it after every rebuild. Virtual veth interfaces need no prefix and no capability.

How-To: Run the test suites

# C++ (Catch2 via ctest) — from boat-platform/
ctest --preset release --output-on-failure
ctest --test-dir build/debug -R TestName --timeout 30 --output-on-failure
ctest --test-dir build/debug -N                      # list tests

# Python SDK + CLI
pytest boat-platform/sdk/python/tests boat-platform/cli/tests -v

# System tests against a live environment
boat test list-environments

Test binaries are named boat_unit_*, boat_integration_*, boat_hil_* (HIL tests need BOAT_HIL_ENABLED=1 and hardware), and boat_determinism_seed — the one that runs the same seed twice and asserts bit-identical output. System-test conventions: test/Structure.md.

FAQ

Do I need a running gateway to use the web pages?
Depends which ones. The gateway UIs (Launcher, Dashboard, Nodes, Commander, Recorder — ports 8080-8086, start_ui.sh) control or observe a live gateway. The standalone tools (PDU Editor, Trace Analyzer, Trace Editor, Eth Analyzer — ports 8087-8090, start_tools.sh) work on files and run fine completely offline; only optional actions like "Push to Gateway" need one.
What's the difference between boat trace replay and boat replay import + start?
boat trace replay reads the file locally and sends each CAN frame via gRPC, paced by the client — simple, but ~5-8 ms per frame and CAN only. boat replay import converts and uploads the trace once; start/ stream then plays it on the gateway's own tick timer with no per-frame overhead, works for Ethernet too, and supports pause/resume/seek and repeated replays. See Replay a CAN trace.
Can one trace file contain both CAN and Ethernet?
Yes — .pcapng (open, Wireshark-readable, the Recorder's default) and BoAt's internal .trace format both carry mixed traffic on one timeline. Classic .pcap cannot (one link type per file), and .asc/.blf are CAN-only. See Mixed traces with PCAPNG.
Frames don't appear on the bus. What do I check first?
  1. Interface up? ip link show vcan0 / ip -d link show type can
  2. Gateway sees it? boat frame list-ifaces (also proves the gRPC connection — UNAVAILABLE means wrong host/port or gateway not running)
  3. Watch the kernel directly: candump vcan0 / tcpdump -i eth0
  4. For Ethernet replay: was --eth-iface set at playback, and is the interface registered via BOAT_ETH_INTERFACES (with raw: + setcap for physical NICs)?
  5. Replay filters (--channel, --id, IP filters) can silently drop everything — try without filters.
What is a .trace file and where do they live?
BoAt's internal binary trace format: a stream of length-prefixed serialized boat.v1.Frame protobuf records — what the replay engine executes directly. Every boat replay import stores one on the gateway at /tmp/<trace_id>.trace, which is why the Trace Editor's file list scans /tmp. It's an internal format: to give a trace to someone outside BoAt, export it to .pcapng (boat replay export or the Trace Editor button).
How do I get an edited trace back into the gateway?
Use the Trace Editor's Push to Gateway button — it uploads the in-memory frames via ImportTraceData. You cannot use boat replay import for this: that command only accepts source formats (.asc/.blf/.pcap/.pcapng) and always re-converts. After pushing: boat replay start --trace <id> --buses vcan0.
CMake says my version is too old.
BoAt needs CMake ≥ 3.24; Ubuntu 22.04 ships 3.22. Install a Kitware binary release or add the Kitware APT repository.
Why does the build want Rust/cargo?
It's a build-time-only transitive dependency of iceoryx2 (the shared-memory IPC library used for payloads > 4 KB). Install via rustup — distro cargo packages are frequently too old. No Rust code exists in BoAt itself.
When do I need python-can?
Only for Vector formats: reading/writing .asc and .blf (recording in those formats, importing them, analyzing them). PCAP and PCAPNG handling is built into BoAt with no third-party dependency. If it's missing you get a clear pip install python-can error exactly where it's needed.
Why does the Launcher need passwordless sudo?
Creating network interfaces (modprobe vcan, ip link add/set) is a privileged operation. The Launcher shells out to these commands when you create vcan/veth interfaces from the browser. Alternatively create the interfaces manually once and never grant the UI sudo.
Why must physical Ethernet use raw: and setcap?
Injecting arbitrary L2 frames on a real NIC requires a raw AF_PACKET socket, which needs CAP_NET_RAW. Grant it once per build: sudo setcap cap_net_raw+ep <gateway binary>. Prefer this over running as root (root-owned files under /tmp break later non-root runs), and remember setcap does not survive rebuilds. Virtual veth pairs need neither.
Are boat can and boat eth gone?
They still work but are deprecated thin wrappers. Since ABI v8 all bus types share one frame model, so boat frame send / boat frame subscribe with --bus-type/--bus-types is the primary interface.
How deterministic is "deterministic"?
Bit-identical: the boat_determinism_seed test runs the same seeded scenario twice and asserts byte-for-byte equal output. This holds as long as plugins don't introduce unseeded randomness or wall-clock dependence — the same rule the core codebase enforces on itself.
Can I open BoAt's trace files in Wireshark?
.pcapng and .pcap — yes, directly, including CAN (Wireshark decodes SocketCAN link-layer frames, classic and FD). The internal .trace format — no; export it first with boat replay export in.trace out.pcapng.
The server-side replay client "hangs" after Replaying…
The client blocks on the event stream while the gateway plays the trace — frames are still hitting the bus (verify with candump/ tcpdump). Use --verbose for per-event output, or Ctrl+C to detach; the replay itself is controlled with boat replay pause/resume/stop.
