BoAt Platform

Automotive SIL/HIL simulation & testing — CAN, CAN FD, and Ethernet on one gRPC gateway.

gRPC / Protobuf CAN · CAN FD · Ethernet AUTOSAR PDU / IpduM Python SDK Physical HIL

Introduction

BoAt (Board Automotive Testing) is an open platform for simulating and testing automotive ECU networks. A single C++ gateway process exposes every bus as a gRPC service — virtual or physical — so you can send frames, subscribe to live data, inject faults, replay traces and orchestrate full simulation scenarios from any language that has a gRPC client.

Key capabilities:

  • CAN & CAN FD — send and receive frames on virtual (SocketCAN) or physical interfaces.
  • Ethernet — virtual multicast buses and physical NICs via raw AF_PACKET sockets.
  • AUTOSAR PDU routing — per-PDU routes over CAN or Ethernet (IP/UDP/IpduM containers).
  • Signal-level routing — cross-bus signal mappings defined in a JSON database.
  • Simulation lifecycle — tick-based state machine (create → start → pause → step → reset).
  • Trace & replay — record to .asc/.blf/.pcap, replay deterministically.
  • Fault injection — signal corruption, packet drops, latency injection.
  • Plugin system — dynamically loaded .so plugins with frame/signal callbacks.
  • AI code generation — generate plugin stubs from plain-language descriptions.

Architecture


  ┌─────────────────────────────────────────────────────┐
  │              BoAt Gateway  (:50051 gRPC)             │
  │                                                     │
  │  FrameService  CanService  PduService  ...          │
  │       │                                             │
  │  FrameSink ──────────── single frame→wire sink      │
  │       │            ▲                                 │
  │  CanBusRegistry  EthernetBusRegistry ── DispatchRx ─┐│
  │       │            │                                ▼│
  │  [vcan0] [vcan1]  [veth0] [raw:enxABC]   plugins    ││
  │                                  [ pdu_router can_tp ││
  │                                    tcp  someip ]  on_frame
  └─────────────────────────────────────────────────────┘
         ▲                      ▲
         │ gRPC                 │ gRPC
  ┌──────┴─────┐        ┌───────┴──────┐
  │ boat CLI   │        │ Python SDK   │
  │ (Typer)    │        │ BoAtClient   │
  └────────────┘        └──────────────┘

The gateway is the only process you need to run. All bus drivers live inside it. Clients — CLI, Python nodes, or any gRPC client — connect on TCP port 50051.

ABI v8: the core owns the stateless transport substrate — the single FrameSink is the only path a frame reaches a bus, and every producer (plugins, replay, FrameService.SendFrame) transmits through it. Inbound frames are dispatched to plugins' on_frame, filtered to the bus types each plugin declares. Plugins own stateful conversations only (TCP, ISO-TP, PDU routing, SOME/IP); even PDU routing is a plugin (pdu_router.so) rather than gateway-core logic.

PDU Router (plugin)

As of ABI v8 the PduRouter is a runtime plugin (pdu_router.so), not gateway-core logic. It registers itself as the IPduRouter service, and PduServiceImpl delegates to it via FindService("pdu_router"). It maps 32-bit PDU IDs to physical transports; routes are added at runtime via ConfigureRoute. When a PDU is sent, the router looks up the route and forwards it as a CAN frame or an Ethernet UDP/IpduM datagram (published through set_frame_publisher). Incoming frames are parsed back to PDU frames and delivered to all active SubscribePdus streams.

Quick Start

  1. Build the gateway 
    cd boat-platform
cmake -S . -B build/debug -DCMAKE_BUILD_TYPE=Debug
cmake --build build/debug -j$(nproc)
  2. Start the gateway (virtual buses) 
    # Create virtual CAN interfaces if not already present
sudo modprobe vcan
sudo ip link add vcan0 type vcan && sudo ip link set vcan0 up

# Start with vcan0 and a virtual Ethernet interface
BOAT_CAN_INTERFACES=vcan0 \
BOAT_ETH_INTERFACES=veth0 \
./build/debug/src/gateway/grpc_gateway/boat_gateway
  3. Install the CLI 
    pip install -e cli/
  4. Send your first CAN frame 
    boat frame send --bus-type can --can-id 0x7B --data DEADBEEF --iface vcan0

Starting the Gateway

The gateway binary is at build/debug/src/gateway/grpc_gateway/boat_gateway after a successful build. All configuration is done via environment variables.

# Minimal: one virtual CAN interface
BOAT_CAN_INTERFACES=vcan0 ./boat_gateway

# With multiple CAN and Ethernet interfaces
BOAT_CAN_INTERFACES=vcan0,vcan1 \
BOAT_ETH_INTERFACES=veth0,veth1 \
./boat_gateway

# With physical USB Ethernet adapters (after granting CAP_NET_RAW, see below)
BOAT_CAN_INTERFACES=vcan0 \
BOAT_ETH_INTERFACES=raw:enx28107b9f2016,raw:enx28107b9f2017 \
./boat_gateway
Permissions Physical Ethernet via raw: requires CAP_NET_RAW. Grant the capability to the binary rather than running the gateway as root: sudo setcap cap_net_raw+ep ./boat_gateway. Root-owned files (e.g. under /tmp) left behind by a sudo-run gateway can block later non-root runs from writing to the same paths, so prefer setcap. Reapply it after every rebuild — the capability is stored on the binary's inode.

Physical Ethernet (Raw Socket)

When a name is prefixed with raw: in BOAT_ETH_INTERFACES, the gateway opens an AF_PACKET / SOCK_RAW socket and binds it to that NIC. This gives the gateway direct L2 access — no IP stack involvement.

USB Ethernet adapters Two USB adapters work well for loopback testing. Assign static IPs before starting the gateway: 
sudo ip addr add 50.50.0.1/24 dev enx28107b9f2016
sudo ip addr add 50.50.0.2/24 dev enx28107b9f2017
sudo ip link set enx28107b9f2016 up
sudo ip link set enx28107b9f2017 up
sudo setcap cap_net_raw+ep ./boat_gateway
Then start with: BOAT_ETH_INTERFACES=raw:enx28107b9f2016,raw:enx28107b9f2017 ./boat_gateway

Static IPs set via ip addr add are not persistent across reboots. To persist them, add entries to your network manager or /etc/network/interfaces.

Gateway startup log confirms successful registration: 

[Gateway] Registered raw Ethernet interface 'enx28107b9f2016'
[Gateway] Registered raw Ethernet interface 'enx28107b9f2017'

If you see Failed to open instead, check permissions and that the interface exists.

CLI — boat can-tp

CAN Transport Protocol (ISO 15765-2) — segments large PDUs across multiple CAN frames with Flow Control (BS/STmin) support. Requires the can_tp.so node plugin to be loaded in the gateway.

boat can-tp configure

Configure an N-SDU session between two CAN IDs (source = this node, target = peer). Both --source-addr and --target-addr default to --nsdu-id for single-ID backward compatibility.

OptionTagDescription
--nsdu-id <hex|dec>reqLogical session identifier
--source-addr <hex|dec>optCAN ID of this node (default: nsdu_id)
--target-addr <hex|dec>optCAN ID of the peer node (default: nsdu_id)
--bs <N>0Block Size to advertise in Flow Control (0 = unlimited)
--stmin <N>0Separation Time in ms to advertise in FC (0–127)
--dlc <N>8CAN DLC: 8 for classic CAN, 64 for CAN FD
--rx-buffer <N>4095RX reassembly buffer size
# Dual-ID session: tester (0x7E0) → ECU (0x7E8)
boat can-tp configure --nsdu-id diag --source-addr 0x7E0 --target-addr 0x7E8 --bs 0 --stmin 0

# Single-ID (backward compat)
boat can-tp configure --nsdu-id 0x7E0

boat can-tp send

OptionTagDescription
--nsdu-id <hex|dec>reqN-SDU ID of the previously configured session
--data <hex>reqPayload hex bytes (will be segmented)
--source-addr <hex|dec>optOverride source CAN ID for this send
--target-addr <hex|dec>optOverride target CAN ID for this send
--bs <N>0Block Size for the session (0 = unlimited)
--stmin <N>0Separation Time in ms
--dlc <N>8CAN DLC: 8 for classic CAN, 64 for CAN FD

CLI — boat pdu

PDU commands manage AUTOSAR-style protocol data units. Before sending a PDU you must configure a route telling the router how to deliver it.

boat pdu route

Adds or updates a routing rule for a PDU ID.

OptionTagDescription
--id <hex|dec>req32-bit PDU ID
--iface <name>reqNetwork interface (tab-completes from /sys/class/net)
--transport <can|eth>reqDelivery transport (tab-completes to can or eth)
--can-id <hex>pdu_idCAN frame ID override. Default: same as PDU ID.
--src-ip <addr>optSource IP (IPv4 dotted or IPv6 colon-hex). Enables IP/UDP/IpduM mode.
--dst-ip <addr>optDestination IP. Required when --src-ip is set.
--src-port <N>0UDP source port
--dst-port <N>0UDP destination port
--ttl <N>64IPv4 TTL / IPv6 Hop Limit
--ethertype <hex>0x88B5Simulation-only EtherType (used when --dst-ip is not set)
--vlan <ID>0802.1Q VLAN ID (0 = untagged)
--send-type <mode>noneTransmission schedule: none | cyclic | onchange | mixed. Requires --cycle-ms etc.
--cycle-ms <N>0Base cycle in ms for cyclic/mixed modes.
--fast-ms <N>0Fast repetition period in ms for OnChange or Mixed modes.
--reps <N>0Number of fast repetitions per change event.
# Route a PDU over CAN
boat pdu route --id 0x00AC0001 --transport can --iface vcan0

# Cyclic schedule: send every 100ms
boat pdu route --id 0x00AC0001 --transport can --iface vcan0 \
               --send-type cyclic --cycle-ms 100

# OnChange schedule: send on change with 3 fast reps at 10ms
boat pdu route --id 0x00AC0002 --transport can --iface vcan0 \
               --send-type onchange --fast-ms 10 --reps 3

# Mixed: cyclic at 200ms + OnChange with 2 fast reps at 20ms
boat pdu route --id 0x00AC0003 --transport can --iface vcan0 \
               --send-type mixed --cycle-ms 200 --fast-ms 20 --reps 2

# Route over UDP/IPv4 — real IP/UDP Ethernet frame
boat pdu route --id 0x00AC0001 --transport eth --iface enx28107b9f2016 \
               --src-ip 50.50.0.1 --dst-ip 50.50.0.2 \
               --src-port 5000 --dst-port 5001

# Simulation-only: bare Ethernet with custom EtherType (no IP)
boat pdu route --id 0x00AC0001 --transport eth --iface veth0 \
               --ethertype 0x88B5

boat pdu remove-route

Removes a PDU routing rule and its transmission schedule entirely.

OptionTagDescription
--id <hex|dec>req32-bit PDU ID to remove
boat pdu remove-route --id 0x00AC0001
Stopping a schedule
Three options: (1) --send-type none keeps the route but stops auto-sends, (2) remove-route deletes the route and schedule entirely, (3) use an I-PDU group to silence the PDU while keeping configuration.
IP mode vs simulation mode
When --src-ip / --dst-ip are set, the gateway builds a real IPv4/IPv6 + UDP + IpduM datagram. When omitted, it uses a lightweight simulation format: [PDU_ID 4 bytes big-endian][payload] wrapped in a custom EtherType frame.

Shell Tab Completion

The CLI has built-in shell completion for bash, zsh, and fish. Install it once:

# Install (adds a source line to your shell rc)
boat --install-completion

# Or print the script to inspect / integrate manually
boat --show-completion

After restarting your shell (or sourcing the rc file), the following complete dynamically:

  • --transport → can, eth
  • --iface → all NICs from /sys/class/net/
  • --db → *.json files in the current directory
  • --msg in boat pdu send/container → ETH_PDU message names from --db
  • --msg in boat db show → all message names from --db
Tip: If you type --db path/to/pdu_db_example.json --msg <TAB>, completion reads that specific database file. If --db hasn't been specified yet, it falls back to pdu_db.json in the current directory.

Python SDK — PduNode

Abstract base class for Python PDU processing nodes. Subclass it, override on_pdu(), and call run() or run_background().

from boat.pdu_node import PduNode
from boat.v1 import pdu_pb2
import socket

class MotorNode(PduNode):
    def on_pdu(self, pdu) -> None:
        print(f"PDU 0x{pdu.pdu_id:08X}  payload={pdu.payload.hex(':')}")

node = MotorNode(pdu_ids=[0x00AC0001, 0x00AC0002])

# Configure a route
def ip(s): return socket.inet_pton(socket.AF_INET, s)
node.configure_route(
    pdu_id=0x00AC0001,
    transport=pdu_pb2.PDU_TRANSPORT_ETHERNET,
    iface="enx28107b9f2016",
    src_ip=ip("50.50.0.1"),
    dst_ip=ip("50.50.0.2"),
    src_port=5000,
    dst_port=5001,
)

# Send a PDU
node.send(0x00AC0001, bytes([0x01, 0x02, 0x03]))

# Run in foreground (blocks, calls on_pdu for each received frame)
node.run()

# Or run in background thread
t = node.run_background()
# ... do other work ...
node.stop()

Method Reference

configure_route(pdu_id, transport, iface, can_id=0, ethertype=0x88B5, vlan_id=0, src_ip=b"", dst_ip=b"", src_port=0, dst_port=0, ttl=64) → bool
Adds or updates a routing rule. Returns True if the gateway accepted it. For real IP delivery set src_ip and dst_ip to 4-byte (IPv4) or 16-byte (IPv6) values.

list_routes() → list

IpduM Container Wire Format

BoAt uses the AUTOSAR IpduM LONG header format. Each PDU entry in a container frame is: 

╔══════════════════╦══════════════════╦═══════════════════════╗
║  PDU_ID (4B BE)  ║  DLC (4B BE)     ║  payload (DLC bytes)  ║
╚══════════════════╩══════════════════╩═══════════════════════╝

Multiple entries are concatenated. The container bytes are then wrapped in:

Ethernet frame:
  [Ethernet Header 14B]
  └─ IPv4/IPv6 header
      └─ UDP header
          └─ IpduM container (one or more entries)

Example — two PDUs in one frame:

PDU 0x00AC0001 with payload 0x0200000000000000 (8 bytes):
  00 AC 00 01  00 00 00 08  02 00 00 00 00 00 00 00

PDU 0x00AC0002 with payload 0xAA (1 byte):
  00 AC 00 02  00 00 00 01  AA

Combined: 00 AC 00 01 00 00 00 08 02 00 00 00 00 00 00 00
          00 AC 00 02 00 00 00 01 AA

How-To: Send a PDU over IP/UDP

Sending a PDU over a real Ethernet link requires two steps: first configure a route, then send.

  1. Configure the route 
    boat pdu route --id 0x00AC0001 --transport eth \
               --iface enx28107b9f2016 \
               --src-ip 50.50.0.1 --dst-ip 50.50.0.2 \
               --src-port 5000 --dst-port 5001
  2. Send the PDU 
    boat pdu send --id 0x00AC0001 --data 0200000000000000
    Or from the database with signal packing: 
    boat pdu send --db config/pdu_db_example.json \
              --msg MotorSpeed_PDU --sig MotorSpeed_ETH=100

The gateway builds a real IPv4 + UDP + IpduM datagram and transmits it via the raw socket driver. The receiving host can parse it as a standard UDP packet with IpduM payload.

How-To: IpduM Container (Multiple PDUs in One Frame)

An IpduM container groups multiple PDU IDs into a single Ethernet UDP frame. When any member PDU is sent, the gateway immediately flushes all slot payloads that have been written at least once.

  1. Register the container (reads member PDUs and IPs from the database) 
    boat pdu container --msg Motor_PDU_Container \
                   --db config/pdu_db_example.json \
                   --iface enx28107b9f2016 \
                   --src-ip 50.50.0.1 --dst-ip 50.50.0.2 \
                   --src-port 5000 --dst-port 5001
  2. Send member PDUs — they are automatically batched 
    # First send: only MotorSpeed_PDU in the frame
boat pdu send --db config/pdu_db_example.json \
              --msg MotorSpeed_PDU --sig MotorSpeed_ETH=100

# Second send: both PDUs in the same frame
boat pdu send --db config/pdu_db_example.json \
              --msg Clamp15_PDU --sig Clamp15_ETH=1
Container vs per-PDU route
Once a container is registered, you do not also need to call boat pdu route for the individual PDU IDs — the container definition overrides the per-PDU route lookup.

I-PDU Groups

Groups enable or disable sets of PDUs at runtime. PDUs in a disabled group are silently dropped — no send, no dispatch to subscribers. Useful for simulating bus-off conditions, silent modes, or domain gating. 

# CLI

Transmission Schedules

Routes can specify automatic sending behavior. The transmission engine is driven by the gateway's tick timer (default 1ms). The tick interval is the minimum achievable cycle time.

ModeCLI valueBehavior
Cyclic--send-type cyclic --cycle-ms NSend every N ms
OnChange--send-type onchange --fast-ms N --reps MSend on payload change, then M fast repetitions at N ms intervals
Mixed--send-type mixed --cycle-ms N --fast-ms M --reps RCyclic at N ms + OnChange with R fast reps at M ms
# Cyclic: every 100ms
boat pdu route --id 0x100 --transport can --iface vcan0 \
               --send-type cyclic --cycle-ms 100

# OnChange: send on change + 3 fast reps at 10ms
boat pdu route --id 0x200 --transport can --iface vcan0 \
               --send-type onchange --fast-ms 10 --reps 3

# Mixed: cyclic at 200ms + change trigger + 2 reps at 20ms
boat pdu route --id 0x300 --transport can --iface vcan0 \
               --send-type mixed --cycle-ms 200 --fast-ms 20 --reps 2

# Seed the payload (first send), then engine drives the rest
boat pdu send --id 0x100 --data 010203

# Stop the schedule
boat pdu route --id 0x100 --transport can --iface vcan0 --send-type none

Signal Packing Reference

The Message.pack() method encodes physical values into raw bytes using a two-step process:

1. Physical → Raw conversion

raw = round((physical - Offset) / Factor)

Example: MotorSpeed with Factor=0.5, Offset=0 at 100 rpm → raw = round(100 / 0.5) = 200

2. Bit insertion

ByteOrderValueBit layout
0 — Intellittle-endianStartPos is the LSB; bits fill toward higher bit numbers, crossing byte boundaries byte-by-byte from low to high.
1 — Motorolabig-endianStartPos is the MSB; bits fill toward lower bit numbers within each byte.

The reverse (raw → physical) formula is: physical = raw * Factor + Offset

FAQ

The gateway starts but boat frame list-ifaces shows no interfaces

The gateway only registers interfaces that exist at startup time. Check:

  • BOAT_CAN_INTERFACES is set correctly and the interface actually exists: ip link show vcan0
  • For vcan: load the kernel module first: sudo modprobe vcan, then create the interface: sudo ip link add vcan0 type vcan && sudo ip link set vcan0 up
boat pdu send fails with RPC error [NOT_FOUND]: No route configured

This means either:

  1. No route has been configured for this PDU ID — run boat pdu route first.
  2. The Ethernet interface failed to register at startup — check the gateway log for [Gateway] Failed to open raw Ethernet interface.
  3. The physical NIC had a transient TX buffer overflow. The gateway retries up to 3 times with 2 ms spacing; if you still see failures, try sending again.

Note: ConfigureRoute succeeds even if the interface is not registered (it only adds an in-memory routing rule). The failure only surfaces when you actually try to send.

I configured a PDU container but the subscriber receives 0 frames

Make sure the receiving side also has a matching route or container configured so the PduRouter recognises incoming frames as belonging to a PDU. Without that, OnEthernetFrame silently drops the packet. The fix: SubscribePdus dispatches to subscribers for any PDU ID that is registered — either via ConfigureRoute or ConfigureContainer — on the receiving gateway.

If you're testing loopback (same gateway sends and receives), the route or container configuration is shared, so this should work automatically after the container is registered.

How do I use physical CAN interfaces (not virtual)?

BoAt uses SocketCAN. Any SocketCAN-compatible adapter works:

For CAN FD, also set dbitrate:

sudo ip link set can0 up type can bitrate 500000 dbitrate 2000000 fd on

      ip4 50.50.0.1/24 gw4 0.0.0.0

How do I run the gateway without sudo for raw Ethernet?

Grant the CAP_NET_RAW capability to the binary — this is the recommended approach, not just a workaround. Running the gateway with sudo instead leaves root-owned files behind (e.g. trace files under /tmp) that block later non-root runs from writing to the same paths.

sudo setcap cap_net_raw+ep ./build/debug/src/gateway/grpc_gateway/boat_gateway

After this, you can run the gateway as a normal user with raw Ethernet support. The capability is stored on the binary's inode, so reapply it after every rebuild.

What is the difference between simulation Ethernet (ETH) and real Ethernet?

The simulation (virtual) path uses a lightweight frame format: [PDU_ID 4 bytes big-endian] + payload wrapped in a custom EtherType (default 0x88B5). No IP or UDP headers are involved — it's a simple simulation convenience.

The real (IP/UDP) path builds a proper IPv4 or IPv6 + UDP + IpduM datagram. Enable it by providing --src-ip and --dst-ip in the route configuration. This is compatible with standard network equipment and AUTOSAR stacks.

Can I connect from a different machine?

Yes. The gateway listens on 0.0.0.0:50051 (all interfaces). Use the gateway's IP address in the CLI:

Or in the Python SDK: BoAtClient("192.168.1.10:50051")

What is the BusService vs SignalService?

BusService is a lightweight always-on publish/subscribe system for named values. It runs independently of any simulation lifecycle. Use it for high-level vehicle signals that any node can publish or subscribe to at any time. Values can be number, string, bool, or bytes.

SignalService is tightly coupled to the simulation tick. Use it for signals that need to be injected at specific simulation ticks, queried historically, or recorded with tick accuracy.

How is the IpduM container format structured?

BoAt uses the AUTOSAR IpduM LONG header format. Each PDU entry occupies:

[PDU_ID: 4 bytes, big-endian]
[DLC:    4 bytes, big-endian]
[payload: DLC bytes]

Entries are concatenated with no padding. The resulting buffer is carried as the UDP payload. Receiving hosts parse the buffer by reading 8-byte headers and advancing by the DLC to find the next entry.

Tab completion doesn't work after boat --install-completion

The completion script is sourced from your shell's RC file (~/.bashrc or ~/.zshrc). You need to reload it:

source ~/.bashrc   # bash
source ~/.zshrc    # zsh

If you use a non-standard shell or a virtualenv, make sure the boat binary on your PATH is the same one that was installed.
