BoAt Test Framework

How-To Guide for Test Engineers

Introduction

The boat-test framework is a Python-based HIL (Hardware-in-the-Loop) and system test framework for the BoAt platform. It lets you write tests that interact with Devices Under Test (DUTs) over CAN, CAN FD, and Ethernet networks — using either real hardware or virtual interfaces.

Tests are hardware-agnostic: the same test runs against virtual interfaces (for CI), physical hardware (for HIL labs), or a mix of both. The only thing that changes is the environment configuration JSON file.

Key idea: Write your test once, run it on any environment. Switch from virtual CAN to physical CAN by changing one config file.

Setup & Installation

Prerequisites

  • Python 3.11+
  • BoAt CLI installed: pip install -e ./cli
  • BoAt SDK installed: pip install -e ./sdk/python
  • A running BoAt gateway (or access to one)

Quick Install

cd /path/to/boat-platform
pip install -e ./sdk/python[dev]
pip install -e ./cli

Virtual CAN (for testing without hardware)

sudo modprobe vcan
sudo ip link add vcan0 type vcan
sudo ip link add vcan1 type vcan
sudo ip link set vcan0 up
sudo ip link set vcan1 up

Your First Test

Here is a minimal test that sends a CAN frame and checks for a response. Save it as my_first_test.py: 

"""Test: engine responds to RPM request."""
import sys
sys.path.insert(0, "path/to/boat-platform")

from boat.test import TestHarness

harness = TestHarness("config/tests/env_virtual.json")
harness.start()
harness.load_pdu_database("config/pdu_db_test.json")

can1 = harness.can_bus("can1")
can2 = harness.can_bus("can2")

try:
    with harness.step(1, "Send RPM request") as step:
        can1.send(0x100, b'\x01\xF4')
        harness.advance(100)
        frame = can2.expect(can_id=0x300, timeout_ms=500)
        step.assert_true(frame is not None)
        step.assert_equal(frame.can_id, 0x300)
finally:
    harness.stop().save("report.json")

Run it:

python3 my_first_test.py
This depends on a running gateway with the configured interfaces. For testing without a gateway, see the pre-flight checks and CLI runner sections.

TestHarness API

The TestHarness is the main entry point for all test scripts. It manages gateway lifecycle, bus connections, DUT interaction, test steps, and reporting.

Lifecycle

from boat.test import TestHarness

harness = TestHarness("config/tests/env_virtual.json")
harness.start()       # starts gateway, connects gRPC, starts trace
# ... run your test ...
report = harness.stop()    # stops trace, shuts down gateway
report.save("report.json")

CAN Bus Operations

TestCanBus provides both raw frame and symbolic signal-level access.

Raw Frame Operations

can1 = harness.can_bus("can1")

# Send a CAN frame
can1.send(can_id=0x100, data=b'\x01\xF4', flags=0)

# Wait for a matching frame (with timeout)
frame = can1.expect(can_id=0x300, timeout_ms=500)

# Subscribe to all frames on this bus
for frame in can1.subscribe():
    print(f"ID=0x{frame.can_id:X} Data={bytes(frame.data).hex()}")

Signal-Level Operations (with PDU Database)

harness.load_pdu_database("config/pdu_db_test.json")
can1 = harness.can_bus("can1")
can2 = harness.can_bus("can2")

# Pack signals and send
can1.send_signal("VehicleSpeed", {"VehicleSpeed": 100.0})

# Receive frame and unpack signals
values = can2.expect_signal("VehicleSpeed",
                            signals={"VehicleSpeed": 100.0},
                            can_id=0x100)
send_signal() and expect_signal() use the PDU database to pack/unpack signals with correct bit positions, byte order (Intel/Motorola), factor/offset scaling, and CAN ID lookup.

Ethernet Operations

eth0 = harness.eth_bus("eth0")

# Send an Ethernet frame
eth0.send(
    dst_mac=b'\x00\x11\x22\x33\x44\x55',
    ethertype=0x88B5,
    payload=b'\x01\x02\x03',
)

# Wait for a frame with matching ethertype
frame = eth0.expect(ethertype=0x88B5, timeout_ms=1000)

PDU Signal Packing

The PduHelper class wraps the PDU database and provides symbolic signal access. You don't need to know CAN IDs, bit positions, factors, or byte orders — everything is read from the database. 

from boat.test import PduHelper

helper = PduHelper("config/pdu_db_test.json")

# Look up a message
msg = helper.get_message("VehicleSpeed")
print(f"CAN ID: 0x{helper.lookup_can_id('VehicleSpeed'):X}")

# Pack signals into raw bytes
payload = helper.pack("VehicleSpeed", {"VehicleSpeed": 100.0})

# Unpack raw bytes into signal values
values = helper.unpack("VehicleSpeed", raw_bytes)
print(values["VehicleSpeed"])  # → 100.0

The PDU database format supports Intel and Motorola byte order, unsigned/signed/float/bool value types, factor/offset scaling, and enum value labels.

Steps & Assertions

Test steps structure your test into logical, reportable units. Each step records stimuli, observations, and assertions in the report. 

with harness.step(1, "Send RPM request") as step:
    step.record_stimulus(type="can", bus="can1", can_id=0x100, data="01F4")

    can1.send(0x100, b'\x01\xF4')
    harness.advance(100)

    frame = can2.expect(can_id=0x300, timeout_ms=500)
    step.record_observation(type="can", bus="can2", can_id=0x300,
                           data=bytes(frame.data).hex())

    # Assertions (auto-recorded in report)
    step.assert_true(frame is not None, "response received")
    step.assert_equal(frame.can_id, 0x300, "correct CAN ID")

Step API

MethodDescription
step.assert_true(condition, expr)Assert condition is truthy
step.assert_equal(actual, expected, expr)Assert equality
step.assert_frame_matches(frame, can_id, data)Assert CAN frame matches
step.record_stimulus(...)Record a stimulus event
step.record_observation(...)Record an observation

Parallel Execution

Run independent tests concurrently with the -n flag. Each test runs in its own subprocess. Reports are generated per test. 

# Run with 4 parallel workers
boat test run manifest.json -n 4

# Stop on first failure in parallel mode
boat test run manifest.json -n 4 --stop-on-failure
Parallel execution is safe because each test is an independent subprocess. Tests must not depend on state from other tests.

Test Matrix

Run the same suite against multiple environment configurations. Each environment gets its own subdirectory under --report-dir. 

# Run against three environments
boat test run manifest.json \
  --matrix "config/tests/env_virtual.json,config/tests/env_hybrid.json"

# Output structure:
# reports/virtual-ci/20260616_*_TC-001/report.html
# reports/hybrid-hil-01/20260616_*_TC-001/report.html

Pre-flight Checks

Validate your environment before running tests. The checks verify that CAN/Ethernet interfaces exist, the gateway binary is present, and the gateway is reachable.

boat test run manifest.json --preflight

HTML Report

Every test run produces a self-contained HTML report (unless --no-html is set). Open it in any browser. The report shows:

  • Test verdict with color-coded badge (PASS green / FAIL red)
  • Test metadata (ID, name, version, description)
  • Execution info (timestamps, duration, runner, CI job)
  • Environment configuration snapshot
  • Precondition steps table
  • Per-step collapsible sections with:
    • Stimuli table (CAN/Ethernet TX frames)
    • Observations table (CAN/Ethernet RX frames)
    • Expected values table
    • Assertions with color-coded rows (PASS green, FAIL red, ERROR orange)
    • Trace references and attachment links
  • Full test summary

Click on a step header to expand/collapse its details.

# Reports are in timestamped folders
reports/20260616_143022_TC-RPM-001/
├── report.html          ← Open this in a browser
├── report.json          ← Full structured data
├── report.junit.xml     ← JUnit for CI
└── stdout.txt           ← Captured output

JUnit XML

Each test also generates a report.junit.xml file in the JUnit format. This is used by CI systems (Jenkins, GitLab CI, GitHub Actions) for test result reporting.

Each step maps to a JUnit <testcase>. Assertion failures become <failure> elements. Step errors become <error> elements.

Trace Recording

The recorder daemon captures raw CAN and Ethernet traffic to files (BLF/ASC/PCAP). Use it when you need to inspect bus traffic after a test run. 

# Start the recorder daemon (separate terminal)
bash start_ui.sh    # or: python3 ui/recorder.py

# Run tests with trace capture
boat test run manifest.json --recorder-url http://localhost:8083

# Trace files appear in the report folder
reports/20260616_143022_TC-RPM-001/
  └── rec_20260616_143022_abcd.blf

C++ Tests

For in-process deterministic tests (no gateway required), use the C++ TestHarness. Tests use Catch2 and run without gRPC. 

#include <boat/test_harness.h>

TEST_CASE("CAN send and capture", "[hil]") {
    boat::test::TestHarness harness;
    auto& can1 = harness.AddCanBus("can1");

    {
        auto step = harness.Step(1, "Send frame");
        can1.Send(0x100, {0x01, 0xF4});
        harness.Advance(std::chrono::milliseconds(100));

        auto& mock = harness.MockCan("can1");
        step.Assert(!mock.written.empty());
        step.Assert(mock.written[0].can_id == 0x100);
    }

    harness.Report().Save("report.json");
}

Add tests to the build with:

add_boat_test(boat_test_my_test test_my_test.cpp)

Run:

cmake --build --preset debug --target boat_test_my_test
./build/debug/src/tests/boat_test_my_test

Best Practices

1. Write Hardware-Agnostic Tests

Reference buses by logical name (can1, can2), not interface names (vcan0, can0). The environment config maps logical to physical.

2. Always Use Steps

Each logical action should be a separate step. Steps with assertions produce structured, readable reports. Without steps, you only get a pass/fail verdict.

3. Load the PDU Database

Use harness.load_pdu_database() to enable symbolic signal access. It saves you from manually computing CAN IDs, bit positions, and factor/offset conversions.

4. Keep Tests Independent

Each test in a manifest should be self-contained. Tests should not depend on state left by previous tests. Use manifest setup / teardown for shared state.

5. Use Pre-flight Checks

Always run --preflight or boat test check-env before a test run to catch configuration issues early.

6. Version Your Tests

Set the version field in your manifest and version in your test case metadata. This helps track changes in reports.

7. Review the HTML Report

The HTML report shows everything: steps, assertions, stimuli, observations, traces. Use it for debugging and as evidence of test results.

8. Run in CI with Virtual Interfaces

Use env_virtual.json for CI. It only needs vcan0, vcan1, and veth0 — no hardware required.

Write test scripts as regular Python scripts, then reference them in a manifest for batch execution with report generation.
